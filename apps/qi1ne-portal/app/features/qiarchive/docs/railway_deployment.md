# Railway Deployment Guide (Option A)

This guide documents the "Identity-First" deployment of QiArchive to Railway. In this architecture, QiArchive (the authority) remains local or in a separate trusted zone, and Paperless (the engine) is hosted on Railway as a downstream consumer.

## Architecture Overview

QiArchive on Railway is deployed as **three separate services** within a single project:

1. **`qiarchive-paperless`**: The main Paperless-ngx engine (built from `deploy/paperless/Dockerfile`).
2. **`qiarchive-postgres`**: A Railway-managed PostgreSQL database.
3. **`qiarchive-redis`**: A Railway-managed Redis instance.

## Deployment Steps

### 1. Provision Databases

- Create a new project in Railway.
- Click **New** -> **Database** -> **Add PostgreSQL**.
- Click **New** -> **Database** -> **Add Redis**.

### 2. Connect the Repository

- Click **New** -> **GitHub Repo** -> Select `QiArchive`.

### 3. Configure the Paperless Service

- In the Service Settings for `qiarchive-paperless`:
  - **Root Directory**: Set to `deploy/paperless/` (or use `RAILWAY_DOCKERFILE_PATH=deploy/paperless/Dockerfile`).
  - **Start Command**: Paperless uses its internal entrypoint; no override is needed.

### 4. Attach a Volume

- Go to the **Volumes** tab in the `qiarchive-paperless` service.
- Add a new volume.
- **Mount Path**: Set to `/paperless`.
- This volume holds your persistent `data`, `media`, and `consume` directories.

## Environment Variable Mapping

The following table maps your local configuration to Railway's managed service variables.

| Local Compose / Env Name | Railway Variable Source | Recommended Railway Variable Name | Notes |
| :--- | :--- | :--- | :--- |
| `POSTGRES_DB` | `qiarchive-postgres` -> `PGDATABASE` | `PAPERLESS_DBNAME` | Usually `railway` by default. |
| `POSTGRES_USER` | `qiarchive-postgres` -> `PGUSER` | `PAPERLESS_DBUSER` | Usually `postgres` by default. |
| `POSTGRES_PASSWORD` | `qiarchive-postgres` -> `PGPASSWORD` | `PAPERLESS_DBPASS` | Automatically generated by Railway. |
| `POSTGRES_HOST` | `qiarchive-postgres` -> `PGHOST` | `PAPERLESS_DBHOST` | Internal hostname (e.g., `postgres.railway.internal`). |
| `POSTGRES_PORT` | `qiarchive-postgres` -> `PGPORT` | `PAPERLESS_DBPORT` | Usually `5432`. |
| `REDIS_URL` | `qiarchive-redis` -> `REDIS_URL` | `PAPERLESS_REDIS` | Full connection string (e.g., `redis://...`). |
| `PAPERLESS_URL` | Railway Public Domain | `PAPERLESS_URL` | The public address of your instance. |
| `PAPERLESS_SECRET_KEY` | Custom Random String | `PAPERLESS_SECRET_KEY` | Generate a fresh secret for production. |
| `PAPERLESS_ADMIN_USER` | Custom | `PAPERLESS_ADMIN_USER` | Admin username for the first run. |
| `PAPERLESS_ADMIN_PASSWORD` | Custom | `PAPERLESS_ADMIN_PASSWORD` | Admin password for the first run. |

## Important Mount Paths

The production `Dockerfile` is pre-configured to look for all persistence at `/paperless/`. Ensure these are protected by the Railway volume:

- `PAPERLESS_DATA_DIR`: `/paperless/data`
- `PAPERLESS_MEDIA_DIR`: `/paperless/media`
- `PAPERLESS_CONSUMPTION_DIR`: `/paperless/consume`

## Handoff from Local

To send the documents from your local `QiArchive` staging to Railway, use the `scripts/ops/push_to_paperless_railway.py` script. It uses the Paperless API, so ensure `PAPERLESS_URL_RAILWAY` and `PAPERLESS_API_TOKEN` are correctly set in your local `.env`.
